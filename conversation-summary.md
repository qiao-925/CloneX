# 会话缩略版（关键结论）

## 主题
今天主要围绕 `git worktree` 的原理、边界、与分支/远程的关系、以及在个人并行开发场景下的最佳实践。

## 关键结论
1. `worktree` 和 `branch` 不是同一个概念。
2. `worktree` 提供的是本地“工作目录隔离”；`branch` 提供的是“历史线隔离”。
3. 一个任务通常建议是“一个分支 + 一个 worktree 目录”。
4. `Publish/Push` 推送的是当前分支，不是把 `worktree` 本身推到远程。
5. 远程仓库不保存 `worktree` 目录关系，只保存提交和分支。
6. 换电脑后可以基于远程分支重新创建本地 `worktree`，从而继续同一任务。
7. `worktree` 解决代码并行问题，但不自动解决端口、数据库、缓存等运行时冲突。
8. 对受保护主分支场景，推荐链路为：任务分支 -> 开发/模块分支 -> `main/master`。

## 实用心智模型
- `branch`: 决定代码历史如何演进。
- `worktree`: 决定你在哪个目录并行工作。
- 远程仓库: 只同步历史与分支，不同步本地目录拓扑。

## 对个人项目的落地建议
1. 主目录保持稳定（集成/验证用途），功能开发尽量放到任务 `worktree`。
2. 每个任务完成后及时提交并按需推送，避免长时间只有本地状态。
3. 若需要跨机器续做，先拉远程分支，再重建对应 `worktree`。
4. 若后续做 GUI 工具，可把“创建/打开/发布/清理 worktree”作为优先功能，“布局同步”作为加分项。
